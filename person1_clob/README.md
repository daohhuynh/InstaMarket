# ⚙️ InstaMarket - Core Engine & Web3 Infrastructure (Person 1)

This directory contains the ultra-low-latency Central Limit Order Book (CLOB), the relational database schemas, and the Solana localnet environment that powers the InstaMarket backend.

**Stack:** C++23 (Engine), Postgres/Supabase (State), Solana/Anchor (Web3).

---

## 1. The Matching Engine (C++)

A highly optimized, zero-allocation matching engine built to ingest the paper trades generated by the AI swarm.

* **Architecture:** O(1) matching latency. Uses flattened, pre-allocated contiguous memory (`std::array`) instead of node-based heaps or maps. Prices strictly range from 1¢ to 99¢.
* **Ingestion:** Custom zero-allocation JSON parser operating directly on the socket buffer to bypass heap fragmentation during swarm spikes.
* **Endpoint:** `http://localhost:8080/api/paper-trades`

### 👉 For Person 2 (Swarm Architect):
The engine is listening for `POST` requests at the endpoint above. 

**Expected Payload Format:**
{
  "market_id": 1,
  "persona_id": 42,
  "price": 60,
  "quantity": 100,
  "side": "YES"
}

**To Build & Run (Locally):**
g++ -std=c++2b -O3 -march=native -flto src/main.cpp -o clob_engine
./clob_engine

## 2. The State Layer (Supabase)

The live Postgres database powering the Chrome Extension DOM. **Note for Demo:** Row Level Security (RLS) is currently disabled so the frontend can read/write without JWT auth friction.

### 👉 For Person 3 (Extension Lead):
You can query these live tables using the standard `@supabase/supabase-js` client:

* `users`: Core user profiles and linked Solana wallet addresses.
* `markets`: The live Polymarket markets (odds, liquidity, resolution state).
* `positions`: The ledger of all trades (both AI personas and the actual user).
* `saved_markets_timeseries`: The historical snapshot table. Use this to render the color-coded deltas (green/red) in the "Save for Later" tab.
